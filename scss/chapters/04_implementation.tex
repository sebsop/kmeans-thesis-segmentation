\section{Implementation and Optimization}\label{sec:implementation}
Achieving real-time 5D K-Means clustering at $120+\text{ FPS}$ \cite{cruden2019simulator, jerald2009scene} requires low-level optimizations tailored to GPU architecture. To ensure deterministic execution, the benchmarking environment is configured with optimized CPU-GPU priority scheduling \cite{microsoft_priority_separation_official, intel_thermal_ultra_gen}, while the compilation and tooling pipeline is managed via CMake and automated analysis tools \cite{cmake_official, clang_tools}. This section presents thread layouts, initialization strategies, compiler unrolling, shared memory caching, and Direct Memory Access (DMA) optimizations.

\subsection{Thread Layout and 2D Coordinate Mapping}
To maximize SM occupancy, the \texttt{StridedData\allowbreak Preprocessor} launches a 2D grid of thread blocks with a block size of $16 \times 16$ ($256$ threads/block), which fits the Blackwell SM occupancy profile \cite{CudaGuide, cook2012cuda}. Each thread calculates its output coordinate:
\begin{align}
x_{\text{out}} &= \text{blockIdx.x} \cdot \text{blockDim.x} + \text{threadIdx.x} \\
y_{\text{out}} &= \text{blockIdx.y} \cdot \text{blockDim.y} + \text{threadIdx.y}
\end{align}
If within bounds, the thread scales the coordinate by the stride $S$ to retrieve the input BGR data, performing normalization and spatial scaling inline \cite{CudaGuide, cook2012cuda}.

\subsection{Centroid Initialization Strategies}
To initialize clustering, the framework supports CPU-bound random selection and a GPU-accelerated K-Means++ seeding algorithm \cite{arthur2007k}. The K-Means++ distance-update steps and cumulative distribution function reduction are offloaded to the GPU using the Thrust library \cite{bell2012thrust}, reducing seeding latency to the sub-millisecond range.

\subsection{Mathematical Intrinsics and Loop Unrolling}
For coordinate distance calculations, we use a templated math helper (\texttt{VectorMath<Dims>}) with the dimension count (5) passed as a template parameter. This forces the compiler to perform complete loop unrolling (\texttt{\#pragma unroll}), expanding the 5D distance loops into sequential assembly operations that eliminate branch prediction overhead \cite{CudaGuide, stroustrup2022tour, cook2012cuda}.

Furthermore, enabling fast math compiler flags (\texttt{-use\_fast\_math}) compiles distance calculations into Fused Multiply-Add (\texttt{\_\_fmaf()}) instructions, executing a multiply and an add in a single clock cycle \cite{CudaGuide}. In the \texttt{QuantumEngine}, we evaluate the cosine phase states using the device intrinsic \texttt{\_\_cosf()}, which executes on the GPU's Special Function Units (SFUs) in a single clock cycle rather than relying on high-precision library calls \cite{cppreference_cos, CudaGuide}. The performance critical loop is shown in Code Snippet~\ref{fig:quantum_sim_code}.

\begin{lstlisting}[float=b, basicstyle=\small\ttfamily, breaklines=true, columns=flexible, frame=single, framextopmargin=3pt, framexbottommargin=3pt, xleftmargin=5pt, xrightmargin=5pt, aboveskip=12pt, belowskip=12pt, captionpos=b, caption={Quantum-inspired phase evaluation in CUDA}, label={fig:quantum_sim_code}]
float target_prob = 1.0f;
#pragma unroll
for (int d = 0; d < 5; ++d) {
    float diff = (p[d] - s_centers[j*5+d])
                 * SCALE_FACTOR * PHASE_OFFSET;
    float cos_val = __cosf(diff);
    target_prob *= (cos_val * cos_val);
}
float dist = 1.0f - target_prob;
\end{lstlisting}

Single-precision floats (FP32) are chosen over FP64 because consumer GPUs offer up to $64:1$ FP32-to-FP64 compute ratios, and FP32 halves memory pressure while exceeding the 8-bit precision requirements of the input BGR channels.

\subsection{Two-Tier Shared Memory Atomics}
Accumulating centroid coordinates from thousands of active threads using global memory atomics causes severe VRAM serialization bottlenecks. To avoid this, a two-tier reduction is implemented:
\begin{enumerate}
    \item \textbf{Block-Local Aggregation:} Threads write their pixel features into fast, on-chip shared memory (\texttt{\_\_shared\_\_}) via atomic operations.
    \item \textbf{Global Writeback:} A single thread per block executes an global atomic addition to write the block's sub-total to VRAM, reducing global memory transactions by a factor of 256 \cite{CudaGuide, cook2012cuda}.
\end{enumerate}

\subsection{Cache Locality and Memory Staging}
An Array of Structures (AoS) format ($[B, G, R, x, y, \dots]$) is utilized for 5D feature storage. Since threads retrieve all five dimensions of a pixel simultaneously for distance evaluation, the first global memory access pulls all subsequent features into the same $L_1$/$L_2$ cache line, ensuring a high cache hit rate \cite{CudaGuide, cook2012cuda}.

Memory transfers across the PCIe bus are optimized by allocating pinned host memory buffers (\texttt{cudaMallocHost}). This enables DMA copies directly to system RAM without involving the CPU. Combining pinned memory with non-blocking asynchronous streams (\texttt{cudaMemcpyAsync}) allows the visual segmented frames to be transferred concurrently while the CPU processes UI updates \cite{CudaGuide, cook2012cuda}. OpenGL texture uploads are synchronized only during screen refresh to prevent flickering.

\subsection{Background Diagnostic Metrics}
To monitor segmentation quality in real-time, the system offloads WCSS \cite{krzanowski1988criterion}, the Davies-Bouldin ($\mathcal{D}\mathcal{B}$) index \cite{davies1979cluster, xiao2017davies}, and the Silhouette Coefficient \cite{rousseeuw1987silhouettes} to a background benchmarking thread. To bypass the $\mathcal{O}(N^2)$ computational bottleneck of the silhouette score on VGA frames, a Monte Carlo approximation selects $M=2,000$ random points \cite{shutaywi2021silhouette}, enabling real-time updates without lagging the rendering pipeline.
